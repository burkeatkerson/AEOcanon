**Most AI crawlers don't run JavaScript, so checking whether they can read your
site means looking at the raw HTML the server returns — not the rendered DOM in
your dev tools.** Two quick tests show you exactly what GPTBot and PerplexityBot
see: disable JavaScript in the browser, and fetch the page with curl.

## Why doesn't 'Inspect Element' tell the truth?

Inspect Element doesn't tell the truth about [crawlability](/glossary/crawlability) because it shows the
*rendered DOM* — the page after your JavaScript has executed and injected content.
A non-rendering crawler never runs that JavaScript, so it sees only the *raw HTML*
the server sent. The two can differ wildly: a [client-rendered](/glossary/client-side-rendering) app often ships an
almost-empty `<div id="root">` that Inspect shows full of content, while View
Source shows it empty. Always test the raw HTML — that's the [access](/pillars/access)
reality.

## How do you run the JS-disabled browser test?

Disable JavaScript in your browser and reload the page — what remains is
approximately what a non-rendering crawler sees.

## How do you check with curl?

Fetch the page with an AI user-agent and read the raw HTML it returns — this is the
exact response the crawler receives, with no rendering involved.

```bash
# See the raw HTML GPTBot receives (first 60 lines)
curl -s -A "GPTBot" https://yourdomain.com/ | head -n 60

# Check whether a specific answer is present in the raw HTML
curl -s -A "GPTBot" https://yourdomain.com/ | grep -i "your answer phrase"

# Compare a few crawlers at once — counts of a key phrase in each response
for ua in "GPTBot" "PerplexityBot" "Bingbot"; do
  count=$(curl -s -A "$ua" https://yourdomain.com/ | grep -ic "your answer phrase")
  echo "$ua: $count match(es)"
done
```

If `grep` finds your answer text, the crawler can read it. If it returns nothing
while the phrase is clearly on the rendered page, your content is being injected by
JavaScript after load — invisible to crawlers that don't render.

<Callout variant="tip" title="Also check the status and headers">
  <p>
    Add <code>-I</code> to see response headers, or <code>-o /dev/null -w
    &quot;%&#123;http_code&#125;\n&quot;</code> to print just the status code. Confirm the
    crawler gets a <strong>200</strong>, not a <strong>403</strong> (blocked), a{" "}
    <strong>302</strong> to a login wall, or a soft <strong>404</strong>. Some
    sites serve bots a different response than browsers — this catches that.
  </p>
</Callout>

```bash
# Confirm the crawler gets a 200, not a block or redirect
curl -s -o /dev/null -w "%{http_code}\n" -A "GPTBot" https://yourdomain.com/
```

## What should you do if the test fails?

If your content fails the test — present in the DOM but missing from View Source
and curl — it's client-rendered, and the fix is to put it in the initial HTML via
[server-side rendering](/glossary/ssr), [static generation](/glossary/ssg), or prerendering. That's covered in [why
JavaScript breaks your AI citation eligibility](/learn/javascript-breaks-ai-citation)
and [how to make a single-page app citable by AI](/learn/make-spa-citable).
